# 舆情分析方法论参考

本文件提供 `sentiment-analysis-methodology` 的完整方法论细则。只有在任务需要更细的语言约束、文书类型区分或证据强度规则时再读取，不要默认把整份内容塞进上下文。

## Language Contract

```json
{
  "positioning": "文本面向高知读者、业务研究人员或传播观察者，重点是解释舆情现象的形成机制、传播结构与现实含义，而不是渲染情绪或堆砌信息。",
  "principles": [
    "事实先行，判断克制，解释充分，表达可读。",
    "核心观点必须由数据、样本、案例或条目证据支撑，避免空泛结论与无证据推断。",
    "文本要符合新媒体阅读习惯，但不能滑向情绪化写作、评论化写作或纯观点输出。"
  ],
  "language_style": [
    "语言准确、简洁、具体，有节奏感，但不夸张、不煽情、不口号化。",
    "允许适度口语化，但不能损害判断的严肃性。",
    "抽象判断要落到具体场景、传播动作、内容形态或用户反应上。"
  ],
  "data_usage": [
    "数据用于支撑论点、限定判断边界、揭示结构变化，不用于数字堆砌。",
    "数据出现时要交代时间区间、平台范围、统计口径、对比对象或结构位置。",
    "若数据只能说明相关性，不能直接上升为因果结论。",
    "一个论证单元通常只保留一处最有代表性的样本配合数据说明。"
  ],
  "judgement_rules": [
    "写作重点是说明为什么会这样、意味着什么、还可能进一步影响什么。",
    "一个段落只承担一种主要功能，不混写走势、原因、风险和建议。",
    "所有判断都应由事实和结构推导，而不是主观宣告。"
  ],
  "analysis_focus": [
    "优先解释传播节点形成、主题切换、情绪变化、平台失衡、主体行为与扩散路径。",
    "强调变量之间如何相互强化，不满足于结果描述。",
    "机制解释优先于表面现象陈述。"
  ],
  "structure_rules": [
    "开头直接切入最具解释价值的矛盾、异常点或关键转折。",
    "正文通常按“现象描述—结构拆解—机制解释—影响判断”推进。",
    "结尾可以回到传播逻辑、治理边界、公共认知或平台机制，但不能脱离证据基础。"
  ],
  "readability_rules": [
    "阅读张力应来自事实反差、结构冲突、预期偏离和传播悖论。",
    "允许适度画面化表达，但必须服务于分析，不能演变为文学化描写。",
    "节奏要有变化，允许用简短过渡句消化复杂信息。"
  ],
  "entity_reference_rules": [
    "除非主体本身就是分析对象，否则不频繁堆叠具体人名、书名或无关出处。",
    "引用案例时优先保留其传播功能和语义特征，而不是机械复述原文。",
    "涉及具体断言时，尽量能回溯到原始条目、样本或官方信息来源。",
    "主体相关判断必须服从原始数据源实际提供的主体维度，不能超出可验证字段范围自行扩写。"
  ],
  "terminology_rules": [
    "术语必须前后一致，同一对象不能频繁更换名称。",
    "主语必须清楚，避免大量无主句、泛指句和官话句。",
    "系统内部字段名、分析模块名、模型名、工具名不应直接暴露在正文中。"
  ],
  "claim_strength_rules": {
    "verified": "已被数据和证据直接支撑的内容，可以写成明确判断。",
    "partial": "证据不完整但趋势较清晰的内容，应写成倾向性判断，并说明条件。",
    "unverified": "无法充分支持的内容，只能写成待核验项、可能性解释或观察线索。",
    "prohibited": [
      "不能把统计相关直接写成因果结论。",
      "不能把聚类标签、关键词共现或单点样本直接写成社会事实。"
    ]
  },
  "avoid_expressions": [
    "避免“值得关注的是”“引发广泛热议”“形成强烈反响”“具有重要意义”等空泛总结。",
    "避免概念堆砌、术语堆砌、模型术语暴露、字段路径引用和过度抽象化陈述。",
    "避免用情绪化、讽刺化、夸饰化语言替代分析。",
    "避免在没有证据支持时使用“表明”“说明”“证明”等高强度判断词。"
  ],
  "execution_rules": [
    "优先完成四项任务：提炼核心判断、围绕判断筛选证据、解释变量关系、控制结论强度与表达边界。",
    "若证据不足，应保留不确定性，不能用语言流畅度掩盖事实空缺。",
    "若存在原始条目、代表性样本或异常案例，应在关键判断处优先吸收其证据价值。"
  ]
}
```

## Style Language Requirements

```json
{
  "monitor_brief": [
    "以监测和提示为主，先说发生了什么、变化在哪里、接下来还要看什么。",
    "段落短、信息密度高，不展开过长机制论证，不预设强建议。",
    "异常波动、平台迁移、主题切换和待跟进事项优先于价值判断。",
    "证据不足的内容写成观察线索，不拔高为结论。"
  ],
  "analysis_report": [
    "以机制解释为主，优先说明为什么会这样、结构如何形成、变量如何联动。",
    "正文按“现象—结构—机制—影响”推进，避免模块结果拼接。",
    "关键判断至少同时绑定统计证据和样本/案例语义，不靠单一指标撑结论。",
    "建议必须交代前提条件和作用路径，不能从风险直接跳到动作。"
  ],
  "evidence_digest": [
    "以证据整理和边界说明为主，不主动扩写强判断或强建议。",
    "优先保留统计口径、样本范围、代表性条目语义和回溯线索。",
    "主题、风险和结论都要落到具体条目或证据组，不做过度解释。",
    "边界、不确定性和证据缺口要明确写出。"
  ]
}
```

## 使用方式

- 默认只依赖 `SKILL.md` 的核心指令。
- 当任务需要更细的语言强度控制时，再读取 `Language Contract`。
- 当任务需要切换文书类型时，再读取 `Style Language Requirements`。
